# 11.소프트웨어 수명주기 관점의 진화하는 아키텍처 구현

- 진화하는 아키텍처란 진화에 방해가 되는 복잡성을 배제하기 위한 것이다.

## 복잡도를 이해하자.

- 강한 결합과 변경 영향도를 최소화하려면 API를 주의 깊게 설계하고 추상화를 적절히 활용해야 한다.

## 진화하는 아키텍처를 위한 설계 원칙

### YAGNI 원칙 : 당장 필요치 않다면 구현하지 말 것

- 그리고 코드를 배포하기 전, 아직 아무도 코드를 사용하지 않은 상황에서 최적화를 수행한다.
- 추상화에는 비용이 따른다.

### 최소 충격 원칙 : 사용자를 놀래키지 말 것

- 학습 곡선이 높아 배우기 어렵거나 이상하게 동작하는 기능은 오히려 사용자를 불안하게 만든다.
- 호출 순서를 문서화하는 것도 좋지만 애당초 호출 순서에 의존하지 않게 하는 편이 더 낫다.
- 구체적이고 긴 메소드 이름이 오히려 동작을 더 잘 설명해주며 이해하기도 쉽다.
- 제약사항을 정확하게 표현할 수 있는 구체적인 타입을 사용하자.

### 도메인 지식은 캡슐화돼야 한다.

- 도메인의 캡슐화는 본질적으로 높은 응집도와 낮은 결합도를 지향하게 된다.
- 도메인 경계를 정의하고 도메인 관련 지식을 캡슐화하는 것은 과학만큼 예술적이다.

## 진화하는 API를 위한 설계 원칙

- API 변경에는 호환성을 유지하자
- 일단 변경이 롤아웃되면, 지원하지 않는 필드를 클라이언트가 얼마나 자주 사용하는지 확인한 후 클라이언트가 업그레이드되거나 지원 일정이 만료될 때 삭제하면 된다.

### 스키마를 사용하자

- 하지만 현실적으로 스키마 없는 방식은 상당한 데이터 무결성 및 복잡도 문제를 지니고 있음이 드러났다. 강력한 타입을 사용하는 스키마 지향 방식은 모호성을 줄여서 애플리케이션의 복잡도를 낮춘다. 단기적으로 간결성을 확보할 수 있다고 해서 모호성이라는 트레이드오프를 감수하는 것은 그다지 좋은 방안이 아니다. 코드와 마찬가지로 데이터 역시 '한 번 기록 여러 번 읽기로 설명할 수 있다. 따라서 스키마를 사용하면 읽기 작업은 더 수월해진다.
