# 04.운영 환경을 고려한 코드 작성

- 코드는 '실제 세계'로 노출되면 이상한 동작을 한다.
- 안전하고 회복성 있는 코딩 기법을 이용해 방어적으로 프로그래밍해서 시스템을 보호해야 한다.

## 장애에 대비하기 위한 방어적 프로그래밍 방안

- 빈값 체크, 입력값 체크

### 예외는 구체적으로 사용하자

- 예외를 애플리케이션 로직에 사용해서는 안 된다.

### 예외는 일찍 던지고 최대한 나중에 처리하자.

- 에러가 발생한 지점으로부터 최대한 가까운 지점에서 예외를 던진다는 뜻이다.
- 어설픈 에러 처리 중 최악인 경우는 처리할 수 없는 예외를 catch 블록 내에서 무시하고 그냥 '삼켜버리는' 것이다. ... 그냥 완전히 무시된 것이다. ... 호출하는 코드가 예외를 던지면 예외를 완전히 처리하거나 혹은 상위 스택으로 전파하자.

```
try {
    // ...
} catch (Exception e) {
    // 에러를 처리할 방법이 없으므로 무시한다
}
```

- 백오프는 비선형으로 대기 시간을 늘리는 방법이다.
- 지터를 추가하면 클라이언트들은 특정 범위에 임의의 값을 백오프 시간에 더한다. 이런 임의성을 추가해서 재시도 작업이 좀 더 넓은 시간대로 퍼지만 요청이 한순간에 몰릴 가능성을 줄일 수 있다.

### 시스템에 멱등성을 부여하자.

- 재시도를 처리하는 가장 좋은 방법은 멱등성있는 시스템을 구현하는 것이다. 멱등성이란 동일한 작업을 여러 번 실행해도 항상 같은 결과가 출력됨을 말한다. 예를 들어 어떤 값을 해시셋에 추가하는 것은 멱등 작업이다.... 원격 API도 클라이언트가 각 요청마다 유일한 ID를 지정하게 하면 멱등성을 구현할 수 있다. 클라이언트가 재시도할 때 실패했던 것과 동일한 요청 ID를 전달하면 된다.

### 리소스를 해제하자.

- 장애가 발생하면 모든 리소스를 해제해야 한다.

## 문제 원인을 찾기 위한 로깅 방안

### 로그는 원자적으로 작성하자

- 로그를 정렬할 때는 시스템 시간에 의존하지 말자. 시스템 시간은 리셋되거나 호스트마다 조금씩 다를 수 있다.

## 설정으로 런타임 동작을 손쉽게 조정하려면

### 설정도 코드처럼 테스트하자

- 설정을 안전하게 변경하려면 설정도 버전 제어 시스템으로 관리하고 리뷰하고 테스트하고 빌드하고 배포해야 한다.
